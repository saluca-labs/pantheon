# Phase 14 Plan 02 — SoulWatch tool_invocation Event Ingestion and API

## Requirements
- **ALETH-02:** SoulWatch ingests tool_invocation events with full telemetry

## Context

**What exists:**
- SoulWatch is a separate FastAPI service at `soulWatch/` with its own database models, pipeline, and routers
- Event ingestion endpoint: `POST /watch/v1/events` in `soulWatch/main.py` -- accepts a JSON dict with `X-Internal-Key` header, passes to `soulWatch/src/pipeline/processor.py:process_event()`
- Pipeline processor runs: anomaly detection -> Sigma rules -> playbooks -> quarantine -> SIEM forward -> WebSocket broadcast
- Existing DB models in `soulWatch/src/database/models.py`: SoulWatchBaseline, SoulWatchAnomaly, SoulWatchDetection, SoulWatchQuarantine, SoulWatchDLQ, SoulWatchCustomRule
- DB connection uses SQLAlchemy async with `soulWatch/src/database/connection.py` (same Base/engine pattern as SoulAuth)
- Existing routers follow the pattern: `APIRouter(prefix="/watch/v1/...", tags=[...])` with `require_permission()` RBAC deps
- The spec (section 3.5.2) defines the `aletheia_tool_invocations` table schema with specific columns and indexes
- tiresias-exec (Plan 01) sends `tool_invocation` events to `POST /watch/v1/events` with the telemetry payload from spec 3.1.5

**What this builds:**
1. A new SQLAlchemy model `AletheiaToolInvocation` matching the spec schema
2. An Alembic migration to create the `aletheia_tool_invocations` table
3. Pipeline extension to recognize `event_type: "tool_invocation"` and persist to the new table
4. REST API endpoints under `/watch/v1/aletheia/tools/` for querying tool invocations (list, detail, agent summary, command frequency)
5. Sigma rule integration so existing detection rules can match on tool invocation events

### Step 3: Extend pipeline processor to handle tool_invocation events

**File:** `soulWatch/src/pipeline/processor.py`

Add a new handler function and integrate it into `process_event()`:

```python
async def _handle_tool_invocation(event: dict, db: AsyncSession) -> dict:
    """Persist a tool_invocation event from tiresias-exec."""
    from soulWatch.src.database.models import AletheiaToolInvocation

    execution = event.get("execution", {})
    policy = event.get("policy", {})
    sanitizer = event.get("sanitizer", {})

    record = AletheiaToolInvocation(
        tenant_id=uuid.UUID(event["tenant_id"]) if event.get("tenant_id") else None,
        invocation_id=event.get("invocation_id", f"inv_{uuid.uuid4().hex[:12]}"),
        agent_id=event.get("agent_id"),
        timestamp=datetime.fromisoformat(event["timestamp"]) if event.get("timestamp") else _now(),
        command=event.get("command", "unknown"),
        args=event.get("args", []),
        full_command=event.get("full_command", ""),
        working_directory=event.get("working_directory"),
        exit_code=execution.get("exit_code"),
        duration_ms=execution.get("duration_ms"),
        stdout_bytes=execution.get("stdout_bytes", 0),
        stderr_bytes=execution.get("stderr_bytes", 0),
        stdout_hash=execution.get("stdout_hash"),
        stderr_hash=execution.get("stderr_hash"),
        policy_verdict=policy.get("verdict"),
        policy_rule_matched=", ".join(policy.get("rules_matched", [])) or None,
        sanitizer_mode=sanitizer.get("mode"),
        sanitizer_verdict=sanitizer.get("verdict"),
        patterns_matched=sanitizer.get("patterns_matched", []),
        environment_hash=event.get("environment_hash"),
    )
    db.add(record)
    return {"persisted": True, "invocation_id": record.invocation_id}
```

In the main `process_event()` function, add routing at the top before anomaly detection:

```python
async def process_event(event: dict, db: AsyncSession) -> dict:
    # Route tool_invocation events to dedicated handler
    event_type = event.get("event_type")
    if event_type == "tool_invocation":
        tool_result = await _handle_tool_invocation(event, db)
        # Still run through detection pipeline for Sigma rule matching
        # (fall through to existing pipeline below)
        result["tool_invocation"] = tool_result

    # ... existing pipeline code (anomaly, sigma, playbooks, etc.)
```

The key design: tool_invocation events are persisted to their own table AND still flow through the existing Sigma/anomaly pipeline so detection rules can match on tool behavior.

### Step 4: Create Aletheia tools API router

**File:** `soulWatch/src/aletheia/__init__.py` (new directory)
**File:** `soulWatch/src/aletheia/router.py`

Create a new router with these endpoints:

#### GET /watch/v1/aletheia/tools/invocations
List tool invocations with filters.

Query params:
- `tenant_id` (required via header X-Tenant-ID or query param)
- `agent_id` (optional filter)
- `command` (optional filter, prefix match)
- `exit_code` (optional filter)
- `policy_verdict` (optional: allow, deny, warn, skipped)
- `sanitizer_verdict` (optional: clean, warn, block, skipped)
- `since` (ISO-8601 timestamp, default: last 24h)
- `until` (ISO-8601 timestamp, default: now)
- `limit` (int, default: 50, max: 500)
- `offset` (int, default: 0)

Response:
```json
{
  "invocations": [...],
  "total": 142,
  "limit": 50,
  "offset": 0
}
```

Each invocation object returns all columns from the model.

#### GET /watch/v1/aletheia/tools/invocations/{invocation_id}
Get a single invocation by invocation_id.

Returns 404 if not found. Returns the full invocation record.

#### GET /watch/v1/aletheia/tools/summary
Aggregate summary for a tenant over a time range.

Query params: `tenant_id`, `agent_id` (optional), `since`, `until`

Response:
```json
{
  "total_invocations": 1542,
  "unique_commands": 23,
  "unique_agents": 5,
  "avg_duration_ms": 287,
  "total_denied": 12,
  "total_sanitizer_blocks": 3,
  "top_commands": [
    {"command": "gws", "count": 823},
    {"command": "gh", "count": 412},
    {"command": "kubectl", "count": 307}
  ],
  "top_agents": [
    {"agent_id": "alfred-main", "count": 987},
    {"agent_id": "codex-runner", "count": 555}
  ],
  "error_rate": 0.034,
  "time_range": {"since": "...", "until": "..."}
}
```

#### GET /watch/v1/aletheia/tools/timeline
Time-bucketed invocation counts for charting.

Query params: `tenant_id`, `agent_id` (optional), `command` (optional), `since`, `until`, `bucket` (1m, 5m, 15m, 1h, 1d -- default 1h)

Response:
```json
{
  "buckets": [
    {"timestamp": "2026-03-21T20:00:00Z", "count": 45, "errors": 2, "denied": 0},
    {"timestamp": "2026-03-21T21:00:00Z", "count": 67, "errors": 1, "denied": 1}
  ],
  "bucket_size": "1h"
}
```

All endpoints require `require_permission("aletheia:read")` RBAC dependency.

### Step 7: Add tool invocation Sigma detection rules

**File:** `soulWatch/src/detection/rules/tool_abuse.yml`

Create Sigma rules that match on tool_invocation events:

```yaml
title: Destructive Tool Command Detected
id: aletheia-tool-001
status: stable
level: high
description: Agent executed a potentially destructive CLI command
logsource:
  product: tiresias
  service: aletheia
detection:
  selection:
    event_type: tool_invocation
    command|contains:
      - "rm -rf"
      - "drop table"
      - "delete --force"
      - "push --force"
  condition: selection
tags:
  - attack.impact
  - aletheia.destructive_tool
response_playbook: investigate
```

```yaml
title: Tool Invocation with Non-Zero Exit Code Burst
id: aletheia-tool-002
status: stable
level: medium
description: Multiple tool invocations failing in rapid succession may indicate misconfiguration or abuse
logsource:
  product: tiresias
  service: aletheia
detection:
  selection:
    event_type: tool_invocation
  filter_failures:
    exit_code|gt: 0
  condition: selection and filter_failures
  timeframe: 5m
  count: 10
tags:
  - aletheia.tool_errors
response_playbook: rate_limit_and_alert
```

```yaml
title: Tool Policy Denial
id: aletheia-tool-003
status: stable
level: high

logsource:
  product: tiresias
  service: aletheia
detection:
  selection:
    event_type: tool_invocation

  condition: selection
```
