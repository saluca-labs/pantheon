# Phase 14 Plan 01 — tiresias-exec Go Binary (CLI Shim)

## Requirements
- **ALETH-01:** tiresias-exec CLI shim captures tool invocations (Go binary, < 10MB, < 5ms overhead)
- **ALETH-03:** tiresias-exec offline mode with local JSONL fallback and background sync

## Context

**What exists:**
- SoulWatch runs as a separate FastAPI service at `soulWatch/` with event ingestion at `POST /watch/v1/events` (accepts JSON event dict, requires `X-Internal-Key` header)
- SoulWatch pipeline processor (`soulWatch/src/pipeline/processor.py`) runs: anomaly detection -> Sigma rules -> playbooks -> quarantine -> SIEM forward -> WebSocket broadcast
- No Go code exists in the repo yet -- this is a new `cmd/tiresias-exec/` directory
- Identity resolution follows the pattern: CLI flags > env vars > config file > JWT token (spec section 3.1.3)
- Telemetry payload is defined in spec section 3.1.5 with `event_type: "tool_invocation"`
- Offline mode defined in spec section 3.1.6: local JSONL, 60s retry with exponential backoff, 100MB rotation

**What this builds:**
A standalone Go binary that wraps any CLI command, captures execution telemetry (exit code, duration, stdout/stderr hashes, byte counts), and reports `tool_invocation` events to SoulWatch. Phase 14 Plan 01 covers the core binary without Action Gate or Response Sanitizer (those are Phase 15+ concerns). The binary must work offline-first: always execute the command, log telemetry locally if SoulWatch is unreachable, sync in background.

### Step 2: Implement main.go -- Entry point and flag parsing

```
Usage: tiresias-exec [flags] -- <command> [args...]
  or:  tiresias-exec <command> [args...]

Flags:
  --agent-id STRING      Override agent identity
  --tenant-id STRING     Override tenant identity
  --soulwatch-url STRING Override SoulWatch endpoint
  --sanitize STRING      Sanitizer mode (passthrough|warn|block) -- reserved for future
  --dry-run              Policy check only, do not execute (future)
  --offline              Force offline mode (skip SoulWatch reporting)
  --config STRING        Config file path (default: ~/.tiresias/agent.yaml)
```

Flow:
1. Parse flags. Everything after `--` (or the first non-flag arg) is the wrapped command.
2. If no command provided, print usage and exit 1.
3. Call `resolveIdentity()` to get agent_id, tenant_id, soulwatch_url, token.
4. Call `executeCommand()` to run the subprocess and capture telemetry.
5. Build the telemetry payload (spec section 3.1.5 format).
6. Call `reportEvent()` async (goroutine). If SoulWatch unreachable, call `writeOffline()`.
7. Write subprocess stdout to os.Stdout, stderr to os.Stderr.
8. Exit with subprocess exit code.

**Performance:** Steps 1-3 must complete in < 1ms. Step 4 is the wrapped command's wall time. Steps 5-7 are < 1ms. Total overhead target: < 5ms.

### Step 3: Implement identity.go -- Agent identity resolution

Priority order per spec section 3.1.3:
1. CLI flags `--agent-id`, `--tenant-id`
2. Environment variables `TIRESIAS_AGENT_ID`, `TIRESIAS_TENANT_ID`
3. Config file `~/.tiresias/agent.yaml` (parsed by config.go)
4. JWT token from `TIRESIAS_TOKEN` env var -- extract `agent_id` and `tenant_id` from claims (base64-decode the payload segment, parse JSON, read `sub` and `tenant_id` fields). Do NOT validate signature -- that is SoulWatch's job.

Return a struct:
```go
type AgentIdentity struct {
    AgentID        string
    TenantID       string
    SoulWatchURL   string
    Token          string
    SanitizeMode   string  // passthrough, warn, block
    OfflineLogPath string
}
```

If agent_id or tenant_id cannot be resolved, log a warning but continue execution. The command must never be blocked by missing identity.

### Step 4: Implement config.go -- YAML config file parsing

Parse `~/.tiresias/agent.yaml`:
```yaml
agent_id: alfred-main
tenant_id: ac6b4247-03ee-4c45-b9ea-06a4aaceeb75
soulwatch_url: https://tiresias.network/watch/v1/events
token: <jwt or soulkey>
sanitize: warn
offline_log: /var/log/tiresias/offline.jsonl
```

Use `gopkg.in/yaml.v3`. If file does not exist, return empty config (not an error). If file is malformed YAML, log warning and return empty config.

### Step 5: Implement executor.go -- Subprocess execution with telemetry

```go
type ExecutionResult struct {
    ExitCode    int
    DurationMs  int64
    StdoutBytes int64
    StderrBytes int64
    StdoutHash  string  // SHA-512 hex
    StderrHash  string  // SHA-512 hex
    Stdout      []byte  // raw output for pass-through
    Stderr      []byte  // raw output for pass-through
}
```

Implementation:
1. `exec.Command(command[0], command[1:]...)` with `cmd.Dir` set to current working directory.
2. Capture stdout and stderr via `cmd.StdoutPipe()` and `cmd.StderrPipe()`. Use `io.TeeReader` to simultaneously hash (crypto/sha512) and buffer the output.
3. Record `time.Now()` before `cmd.Start()`, compute duration after `cmd.Wait()`.
4. If the command is not found (exec.ErrNotFound), print error to stderr and exit 127 (standard shell behavior).
5. If the command is killed by signal, map to appropriate exit code (128 + signal number).
6. Stdout/stderr buffers are written to os.Stdout/os.Stderr after capture completes.

**Key constraint:** The subprocess stdout must be passed through to the caller's stdout byte-for-byte. tiresias-exec must be transparent -- the calling agent should not be able to distinguish `tiresias-exec foo` from `foo` (except for the < 5ms overhead).

### Step 6: Implement reporter.go -- Async SoulWatch event reporting

Build the telemetry JSON payload per spec section 3.1.5:
```json
{
  "event_type": "tool_invocation",
  "version": "1.0",
  "timestamp": "<ISO-8601>",
  "agent_id": "<from identity>",
  "tenant_id": "<from identity>",
  "invocation_id": "inv_<random 12 hex chars>",
  "command": "<first arg>",
  "args": ["<remaining args>"],
  "full_command": "<joined command string>",
  "working_directory": "<cwd>",
  "environment_hash": "<SHA-256 of sorted env var keys>",
  "execution": {
    "exit_code": 0,
    "duration_ms": 342,
    "stdout_bytes": 4821,
    "stderr_bytes": 0,
    "stdout_hash": "sha512:def456...",
    "stderr_hash": "sha512:000000..."
  },
  "policy": {
    "evaluated": false,
    "verdict": "skipped",
    "rules_matched": []
  },
  "sanitizer": {
    "mode": "passthrough",
    "verdict": "skipped",
    "patterns_matched": [],
    "scan_duration_ms": 0
  }
}
```

Send via `POST` to `{soulwatch_url}` with headers:
- `Content-Type: application/json`
- `X-Internal-Key: <token>` (from identity)

Use `http.Client` with 5-second timeout. Fire in a goroutine. If the POST fails (network error, non-2xx status), call `writeOffline()` with the payload. Do NOT block the main goroutine -- the binary must exit immediately after writing stdout/stderr.

Use `sync.WaitGroup` to ensure the reporting goroutine completes before process exit (up to 2-second grace period, then abandon).

### Step 7: Implement offline.go -- Local JSONL fallback and background sync

**Write path (`writeOffline`):**
1. Open (or create) the offline log file at `identity.OfflineLogPath` (default: `~/.tiresias/offline.jsonl`).
2. Append the telemetry JSON as a single line (JSON + newline).
3. Use file locking (or `os.O_APPEND` atomic writes) to prevent corruption from concurrent tiresias-exec invocations.
4. After write, check file size. If > 100MB, truncate oldest entries (read all, drop first 20%, rewrite). This is best-effort -- if truncation fails, log warning and continue.

**Sync path (`syncOffline`) -- background goroutine:**
This runs only when `tiresias-exec` is invoked (not as a daemon). On each invocation:
1. Check if offline log exists and has content.
2. If yes, spawn a goroutine that reads up to 100 entries, POSTs each to SoulWatch.
3. Successfully synced entries are removed from the file (rewrite without them).
4. Retry interval: 60s initial, exponential backoff up to 15 minutes, with jitter.
5. The sync goroutine gets the same 2-second grace period as the reporter.

**Note:** For v1, the sync is opportunistic -- it runs piggy-backed on actual tiresias-exec invocations. A standalone `tiresias-exec sync` subcommand for manual drain is a future enhancement.

### Step 8: Build and verify binary size

Add a `Makefile` at `cmd/tiresias-exec/Makefile`:

```makefile
BINARY=tiresias-exec
VERSION=0.1.0

.PHONY: build build-all clean test

build:
	CGO_ENABLED=0 go build -ldflags="-s -w -X main.Version=$(VERSION)" -o $(BINARY) .

build-all:
	GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build -ldflags="-s -w" -o $(BINARY)-linux-amd64 .
	GOOS=darwin GOARCH=amd64 CGO_ENABLED=0 go build -ldflags="-s -w" -o $(BINARY)-darwin-amd64 .
	GOOS=darwin GOARCH=arm64 CGO_ENABLED=0 go build -ldflags="-s -w" -o $(BINARY)-darwin-arm64 .
	GOOS=windows GOARCH=amd64 CGO_ENABLED=0 go build -ldflags="-s -w" -o $(BINARY)-windows-amd64.exe .

clean:
	rm -f $(BINARY) $(BINARY)-*

test:
	go test -v -race ./...
```

Build with `CGO_ENABLED=0` for static binary. Use `-ldflags="-s -w"` to strip debug info and reduce size.

## Verification

### ALETH-01 verification:
```bash
cd /home/cristian/tiresias/cmd/tiresias-exec
go build -ldflags="-s -w" -o tiresias-exec .

# Binary size < 10MB
ls -la tiresias-exec | awk '{print $5}'  # must be < 10485760

# Transparent execution -- exit code pass-through
./tiresias-exec --offline echo "hello world"
# stdout: hello world, exit code: 0

./tiresias-exec --offline false
# exit code: 1

# Overhead measurement (< 5ms)
time ./tiresias-exec --offline true
# real time should be < 0.010s

# Telemetry capture -- check offline log
./tiresias-exec --offline --config /dev/null echo "test"
cat ~/.tiresias/offline.jsonl | python3 -m json.tool | head -20
# Should contain event_type: tool_invocation, command: echo, execution block with exit_code/duration/hashes
```

### ALETH-03 verification:
```bash
# Offline fallback -- SoulWatch unreachable
export TIRESIAS_AGENT_ID=test-agent
export TIRESIAS_TENANT_ID=test-tenant
./tiresias-exec --soulwatch-url http://localhost:99999 echo "offline test"
# Command executes successfully (exit 0)
# Offline log written:
cat ~/.tiresias/offline.jsonl | tail -1 | python3 -m json.tool
# Should contain the event with agent_id: test-agent

# Multiple offline events accumulate
./tiresias-exec --soulwatch-url http://localhost:99999 date
./tiresias-exec --soulwatch-url http://localhost:99999 whoami
wc -l ~/.tiresias/offline.jsonl  # should be 3+ lines

# Config file parsing
mkdir -p ~/.tiresias
cat > ~/.tiresias/agent.yaml << EOF
agent_id: alfred-main
tenant_id: ac6b4247-03ee-4c45-b9ea-06a4aaceeb75
soulwatch_url: https://tiresias.network/watch/v1/events
offline_log: /tmp/tiresias-test-offline.jsonl
EOF
./tiresias-exec --offline ls /tmp
cat /tmp/tiresias-test-offline.jsonl | python3 -m json.tool | grep agent_id
# Should show: alfred-main
```
