## 1. Overview

The Tiresias Partner Program enables two classes of partners -- Resellers and MSSP Partners -- to build recurring revenue by selling or operating Tiresias for their customers. This spec covers partner registration, approval, tenant provisioning, billing, rev-share, and the enforcement constraints that prevent hierarchy abuse.

### What Exists Today

| Component | Status | Location |
|---|---|---|
| `partner_id` claim in license JWT | Exists but unused downstream | `src/license/validator.py` L10, L43-44 |
| MSSP parent-child hierarchy | Built, max depth 3 | `src/mssp/isolation.py` |
| Tier-gated feature registry | Built | `src/middleware/feature_gate.py` |
| Stripe webhook handler | Built | `src/saas/billing.py` |
| Stripe subscription upgrade | Built | `src/billing/upgrade.py` |
| White-label branding engine | Built, gated to mssp+saas | Portal + API |
| License JWT issuance script | Built, missing `--partner-id` flag | `scripts/issue_license.py` |

**Key gap:** `SoulTenant` in `src/database/models.py` does not declare `parent_tenant_id` or `hierarchy_depth` columns, although `src/mssp/isolation.py` and `src/mssp/router.py` reference them. These columns must be added to the model before any partner work begins.

## 2. Partner Types

### 2.1 Reseller -- To Build

A Reseller sells Tiresias licenses to end customers and earns a recurring rev-share on monthly recurring revenue (MRR) generated by those customers.

| Property | Value |
|---|---|
| Allowed customer tiers | community, starter, pro, enterprise |
| Cannot sell | mssp, saas |
| Own tenant tier | pro (default) or enterprise |
| Tenant provisioning | No -- customers sign up independently; linked by `partner_id` |
| White-label | No |
| Rev-share | Percentage of referred customer MRR |

Resellers do not create sub-tenants. Instead, their referred customers register through a partner referral link (`?ref=PARTNER_CODE`). The customer's `SoulTenant` record is tagged with the reseller's `partner_id`, and all billing flows through the customer's own Stripe subscription. The reseller earns a rev-share on that subscription.

### 2.2 MSSP Partner -- To Build

An MSSP Partner operates a managed security practice on top of Tiresias. They provision and manage tenants for their end customers.

| Property | Value |
|---|---|
| Own tenant tier | mssp |
| Sub-tenant max tier | enterprise (HARD LIMIT) |
| Sub-tenant cannot be | mssp or saas (HARD LIMIT) |
| Max hierarchy depth | 2 levels under partner (partner -> enterprise tenant, no further nesting) |
| Tenant provisioning | Yes -- via API and portal |
| White-label | Yes |
| Rev-share | Percentage of sub-tenant MRR, OR partner bills sub-tenants directly |

```
  MSSP Partner (tier: mssp)
    |
    +-- Customer A (tier: enterprise)    <-- allowed
    +-- Customer B (tier: pro)           <-- allowed
    +-- Customer C (tier: starter)       <-- allowed
    +-- Customer D (tier: mssp)          <-- BLOCKED (hard constraint)
    +-- Customer E (tier: saas)          <-- BLOCKED (hard constraint)
         |
         +-- Sub-sub-tenant             <-- BLOCKED (depth > 2 under partner)
```

### 3.3 Approval Process (Phase 1: Manual)

1. Application stored in `_partner_applications` table with status `pending`.
2. Slack notification to `#partner-ops` channel via webhook.
3. Email notification to `partners@saluca.com`.
4. Admin reviews application in Tiresias admin panel or via API.
5. Admin calls `POST /v1/admin/partners/{id}/approve` with optional overrides (rev-share %, tier).
6. System creates:
   - `_soul_tenants` row with tier `mssp` (MSSP) or `pro` (Reseller)
   - `_partners` row linking `partner_id` to `tenant_id`
   - License JWT with `partner_id` claim
   - Admin soulkey for the partner tenant
   - Rev-share configuration row in `_partner_revshare_config`
7. Welcome email sent with: portal login link, API key, partner code, onboarding docs link.

### 3.4 Rejection

Admin calls `POST /v1/admin/partners/{id}/reject` with a reason. Application status set to `rejected`. Notification email sent to applicant.

## 5. Tier Constraints

**Status:** To Build -- CRITICAL PATH

These constraints are the most important correctness requirement in the partner program. A single violation (an MSSP partner creating an MSSP sub-tenant) would break the billing model and hierarchy assumptions.

### 5.1 Constraint Matrix

| Actor | Can Create Tenant? | Allowed Sub-Tenant Tiers | Max Depth Below Actor |
|---|---|---|---|
| MSSP Partner | Yes | community, starter, pro, enterprise | 1 (flat children only) |
| Reseller | No (refers only) | N/A -- customers choose own tier | N/A |
| Enterprise tenant (under MSSP) | No | N/A | 0 |
| Direct enterprise customer | No | N/A | 0 |

### 5.2 Enforcement Points

# --- ADD to src/mssp/isolation.py ---
async def validate_partner_child_creation(
    db: AsyncSession,
    parent_tenant_id: uuid.UUID,
    requested_tier: str,
) -> None:
    """
    Enforce partner-specific constraints on child tenant creation.
    Raises HTTPException if:
      - requested_tier is mssp or saas
      - parent is itself a child of a partner (depth > 1 under partner root)
    """
    BLOCKED_TIERS = {"mssp", "saas"}
    if requested_tier in BLOCKED_TIERS:
        raise HTTPException(
            status_code=422,
            detail=f"Tier '{requested_tier}' cannot be assigned to partner sub-tenants.",
        )

    # Check if parent is already a child (not a root partner tenant)
    parent_depth = await get_hierarchy_depth(db, parent_tenant_id)
    if parent_depth >= 2:  # partner root is depth 0 or 1
        raise HTTPException(
            status_code=422,
            detail="Cannot create sub-tenants more than 1 level below the partner.",
        )
```

## 6. Stripe Billing & Rev-Share

**Status:** To Build

### 6.1 Billing Models

There are two viable billing models for partner sub-tenants. Both are specified; the recommendation is to implement Model A first.

#### Model A: Partner-Billed (Recommended for Phase 1)

The partner is the Stripe customer. Sub-tenant usage is metered on the partner's subscription.

```
  Stripe
    |
    +-- Partner subscription ($2,499/mo MSSP base)
    |     |
    |     +-- Metered item: sub-tenant count x $199/tenant
    |     +-- Metered item: agent overage (if applicable)
    |
    +-- Partner pays single invoice
    |
    +-- No rev-share needed (partner sets own prices to end customers)
```

**Pros:** Simple. One Stripe customer. Partner handles their own end-customer billing. No rev-share complexity.
**Cons:** Tiresias has no direct billing relationship with end customers. If partner churns, all sub-tenants lose access.

**Implementation:**
- Partner's Stripe subscription includes metered billing items for sub-tenant count.
- Each time `POST /v1/partners/tenants` provisions a new tenant, increment the metered usage.
- Monthly reconciliation cron job verifies metered count matches actual tenant count.
- File: Extend `src/saas/billing.py` with `report_metered_usage()` function.

#### Model B: Direct-Billed with Rev-Share (Phase 2)

Each sub-tenant has their own Stripe subscription. The partner earns a percentage of each sub-tenant's MRR.

```
  Stripe
    |
    +-- Partner subscription ($2,499/mo MSSP base)
    |
    +-- Sub-tenant A subscription ($799/mo enterprise)
    |     +-- 25% rev-share -> $199.75/mo to partner
    |
    +-- Sub-tenant B subscription ($199/mo pro)
    |     +-- 25% rev-share -> $49.75/mo to partner
    |
    +-- Stripe Connect: automated monthly transfers to partner
```

**Pros:** Tiresias retains direct billing relationship. Sub-tenants survive partner churn. Revenue visibility.
**Cons:** Requires Stripe Connect. More complex reconciliation. Partner onboarding requires bank account setup.

**Implementation:**
- Each partner onboards to Stripe Connect (Standard or Express account).
- Sub-tenant subscriptions include `metadata.partner_id`.
- Monthly cron calculates rev-share per partner and creates Stripe Transfer.
- File: New module `src/partners/revshare.py`.

### 6.2 Rev-Share Configuration

| Parameter | Default | Range | Notes |
|---|---|---|---|
| `revshare_pct` | 25 | 10-40 | Percentage of sub-tenant MRR |
| `revshare_model` | `partner_billed` | `partner_billed`, `direct_revshare` | Billing model |
| `payout_frequency` | `monthly` | `monthly`, `quarterly` | When payouts are calculated |
| `payout_min_threshold_cents` | 5000 | 1000-50000 | Minimum payout amount ($50 default) |
| `revshare_effective_date` | signup date | date | When rev-share starts accruing |

These are stored per-partner in `_partner_revshare_config` and can be overridden by admin at approval time.

### 6.3 Monthly Reconciliation Logic

```
  1. On 1st of month at 00:00 UTC, cron triggers reconciliation
  2. For each active partner:
     a. Query all sub-tenants (MSSP) or referred customers (Reseller)
     b. For each, fetch Stripe subscription MRR from metadata or Stripe API
     c. Calculate: revshare_amount = sub_tenant_mrr * (revshare_pct / 100)
     d. Sum all revshare_amounts for the partner
     e. If sum >= payout_min_threshold_cents:
        - Create _partner_payout_ledger entry (status: pending)
        - Phase 1: Flag for manual payout
        - Phase 2: Create Stripe Transfer via Connect
     f. If sum < threshold: roll over to next month
  3. Log reconciliation results, alert on anomalies
```

File: `src/partners/reconciliation.py` (new)

```

### 7.4 How `partner_id` Links to Existing Schema

- **MSSP model:** Partner's `_partners.tenant_id` -> `_soul_tenants.id` (the MSSP root). Sub-tenants have `_soul_tenants.parent_tenant_id` pointing to this root. No separate `partner_id` column needed on sub-tenants; the hierarchy relationship is sufficient.
- **Reseller model:** Referred customers have `_soul_tenants.partner_id` set to `_partners.id`. This is also stored in the license JWT `partner_id` claim for upstream relay.
- **Soulkeys:** `_soulkeys.tenant_id` -> `_soul_tenants.id`. No changes needed. Partner isolation works by scoping to the tenant subtree.

| Step | Migration | Depends On |
|---|---|---|
| 1 | `001_add_tenant_hierarchy_columns.sql` -- add `parent_tenant_id`, `hierarchy_depth`, `partner_id` to `_soul_tenants` | Nothing |
| 2 | Update `src/database/models.py` `SoulTenant` class | Step 1 |
| 3 | `002_create_partner_tables.sql` -- create all 6 new tables | Step 1 |
| 4 | Add SQLAlchemy models for new tables in `src/partners/models.py` | Step 3 |

## 11. Security Considerations

### 11.1 Partner Isolation

Partners MUST NOT be able to see, access, or enumerate each other's tenants.

- All partner API endpoints filter by the authenticated partner's `partner_id` or `tenant_id`.
- The existing `src/mssp/isolation.py` subtree logic already enforces this for MSSP queries.
- Reseller queries use `WHERE partner_id = <caller_partner_id>` in all tenant lookups.
- Admin endpoints (`/v1/admin/partners/*`) are the only paths that can see all partners.

### 11.2 Server-Side Tier Enforcement

Client-side tier selectors are cosmetic. All tier constraints are enforced server-side at:
1. `POST /v1/partners/tenants` -- partner provisioning endpoint
2. `POST /v1/mssp/tenants` -- existing MSSP provisioning endpoint
3. `src/billing/upgrade.py` -- tier upgrade path
4. `src/saas/billing.py` -- Stripe webhook tier updates

Any request that attempts to set a partner sub-tenant to `mssp` or `saas` tier returns HTTP 422 and logs to `_partner_audit_log`.

### 11.3 API Key Scoping

Partner soulkeys are scoped to their own tenant subtree:
- MSSP partners: soulkey's `tenant_id` matches the MSSP root tenant. All subtree queries use this as the root.
- Resellers: soulkey's `tenant_id` matches the reseller's own tenant. They can only read (not modify) referred customer data via the partner API.
- A partner soulkey CANNOT authenticate as a different partner or access tenants outside their subtree.

### 11.4 Audit Logging

All partner operations are logged to `_partner_audit_log`:
- Tenant creation, suspension, reactivation
- Configuration changes (rev-share, white-label)
- API key rotation
- Payout events
- Failed constraint violations (attempted mssp sub-tenant creation)

Log entries include: `partner_id`, `actor_id` (soulkey), `action`, `target_type`, `target_id`, `details` (JSON), `ip_address`, `timestamp`.

### 11.5 Rate Limiting

| Endpoint | Limit | Scope |
|---|---|---|
| `POST /v1/partners/apply` | 3/hour | Per IP |
| `POST /v1/partners/tenants` | 10/hour | Per partner soulkey |
| `GET /v1/partners/*` | 60/minute | Per partner soulkey |

Rate limits are enforced in the API middleware layer. Partner provisioning has a deliberately low limit to prevent abuse (e.g., scripted mass tenant creation).

### 11.6 Data Access Controls

| Actor | Can Read | Can Write |
|---|---|---|
| MSSP Partner | Own tenant + subtree tenants | Create/suspend subtree tenants, white-label config |
| Reseller | Own tenant + referred customer summary (tier, MRR, status) | Nothing on customer tenants |
| Admin | All partners, all tenants | Approve/reject apps, modify config, void payouts |
| Sub-tenant user | Own tenant only | Own tenant only (no awareness of parent partner) |

Sub-tenant users have no visibility into the partner hierarchy. They see their own tenant as a standalone Tiresias deployment.

### 11.7 Secret Handling

- Partner soulkeys follow the same issuance and hashing pattern as all soulkeys (`src/auth/soulkey.py`). The raw key is shown once at creation and never stored.
- Stripe Connect account tokens are stored in `_partners.metadata` (encrypted at rest via database-level encryption).
- Partner API responses never include raw soulkeys except at initial issuance.
- Payout amounts in audit logs use cents (integers) to avoid floating-point issues.
